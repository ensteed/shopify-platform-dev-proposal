* Introduction

Thanks for the platform brief. I have read it in full, including the open questions sections at the end of each module.

My overall read: this is not a Shopify customization project. It is a purpose-built application platform that uses Shopify as a commerce engine and integration point. The scope, the identity architecture, the Klaviyo event pipeline, the four feature modules, and the TapCart delivery requirements all point to a system that needs to be designed and built as a platform from the ground up.

This proposal outlines how I would approach that, what I would build and in what order, what I would charge, and what I need resolved before implementation begins.

* My Understanding of the Project

You are building a custom headless customer portal and content platform layered on top of your existing Shopify Plus, and want to investigate creating an additional TapCart stack. The platform has four feature modules: a digital pattern library, a gallery submission system, a blog comment system, and an online event system.

The architectural core requirement --- and the one that shapes every other decision --- is platform independence. Shopify should remain the order and commerce record, but the application data layer must live in your own system, with Shopify customer IDs treated as external linked identifiers rather than primary keys. Every feature module, every customer record, every piece of community content must be portable.

This requirement means that the database and API layer come first, Shopify integration is a connection layer on top, and all four modules are built against your own platform's data model.

The four modules share infrastructure: a single internal user model, a shared comment component, a unified Klaviyo event pipeline, and a notification preference system. These cross-cutting systems need to be designed once and built correctly, not re-implemented per module.

* Technical Approach

** Stack Recommendation

Your suggested stack direction is sound and I would follow it closely:

- *Backend:* PostgreSQL for all application data. A single REST API serving both the web portal and TapCart.
#+BEGIN_COMMENT
NOTE: The "single API" part matters for TapCart. All four modules get their data from YOUR server, not Shopify. TapCart loads your custom features by calling your REST API via a WebView, same as the web portal does in a browser. Shopify is only contacted server-side for auth verification, fulfillment calls, and product catalog sync.
#+END_COMMENT
- *Webhook layer:* Dedicated listener service for Shopify order and fulfillment events, decoupled from the main API.
#+BEGIN_COMMENT
NOTE: Shopify sends HTTP POST payloads to a URL you register in the Shopify admin (e.g. POST /webhooks/shopify). Each request includes an HMAC-SHA256 signature computed using your webhook secret --- you verify this before processing any payload. "Decoupled from the main API" means the webhook handler is a separate route or process so a burst of incoming webhook events does not affect customer-facing API response times.
#+END_COMMENT
- *Klaviyo:* Event emission built into the service layer as a first-class concern, not bolted on at the controller level.
#+BEGIN_COMMENT
NOTE: Klaviyo's Track API is a simple HTTP POST: event name + customer email + a properties JSON object. "Service layer" means you create a thin KlaviyoService wrapper class with named methods (e.g. trackPatternPurchased, trackCommentReply) rather than scattering raw HTTP calls across your codebase. There will be roughly 15 distinct event types across all four modules, so this abstraction pays off quickly. It also makes Klaviyo easy to stub in tests.
#+END_COMMENT
- *File management:* All pattern file URLs stored as a managed field with a redirect/proxy layer, so the underlying host can change without breaking any customer's library access.
#+BEGIN_COMMENT
NOTE: Your download endpoint reads the file_url column from the DB and issues a 302 redirect. The customer's browser follows it to the actual hosted PDF. Since every download goes through your server first, changing the hosting location is just a DB field update. Customers and Klaviyo emails always use your stable /api/patterns/:id/download URL, never the hosted file URL directly.
#+END_COMMENT
- *Image storage:* S3-compatible object storage (S3 or Cloudflare R2) for gallery photo uploads, with server-side resizing and moderation at upload time rather than on delivery.
#+BEGIN_COMMENT
NOTE: This is specifically for Phase 3 gallery photos (user-submitted images), not the pattern PDFs. Your server receives multipart uploads, runs them through an image processing library (e.g. "sharp" for Node.js) to generate a display-size version and thumbnail, then uploads both to S3/R2. Cloudflare R2 is S3-API-compatible (same SDK, zero code change) but has zero egress fees vs S3's ~$0.09/GB, which matters for frequently served images.
#+END_COMMENT

** Addressing the Open Questions in the Brief

The brief flags a number of specific technical decisions. Here are my positions on each.

*File URL abstraction (Module 1):* I recommend storing a stable internal URL per pattern that proxies to the actual hosted file. This means the download link customers and Klaviyo emails receive never changes regardless of where the file is hosted. When you move hosting, you update one field in the admin panel. No order history breaks, no emails become invalid.

*Historical order scan rate limits (Module 1):* Shopify's REST Admin API supports cursor-based pagination and allows up to 2 requests per second on Plus. For a large order history, I would run this as a background job with conservative pacing (well under the rate limit), checkpointing progress to the database so it can resume safely if interrupted. This is a one-time operation and does not need to be fast --- it needs to be reliable and auditable.

*Auto-fulfill standalone patterns vs. mixed orders (Module 1):* Shopify's Fulfillment API supports fulfilling individual line items within an order. The correct approach is to create a fulfillment for only the digital line item, leaving the physical portion open. The pattern type flag on the pattern record controls whether this auto-fulfillment fires.

*Notification preference storage (Module 2):* I recommend storing preferences in your own database and syncing them to Klaviyo as profile properties. This gives you the source of truth in your own system (portable, queryable, not dependent on Klaviyo's data model), while Klaviyo flows can branch on those properties at send time. You get the best of both: your own platform owns the record, Klaviyo owns the delivery logic.

*TapCart push notification tokens (Module 2):* This is unresolved and needs to be verified with TapCart directly before the frontend architecture is finalized. See the TapCart section below.

*Gallery widget injection (Module 3):* I recommend a Shopify Theme App Extension over script tag injection. Theme app extensions are the current Shopify standard for Shopify Plus, they are version-controlled, they respect the Online Store 2.0 section architecture, and they do not risk being blocked or delayed by theme load order. The gallery data is served from your API regardless --- the extension is only the mounting point.

*Free event registration (Module 4):* I would keep free registrations entirely in your custom database. Creating a zero-dollar Shopify order adds operational noise with no benefit --- it does not improve reporting for free events and it couples your registration data to Shopify's order model unnecessarily, which cuts against your portability requirement.

*VOD access expiry (Module 4):* Soft-hide is the right call. The data record is retained, the expiry timestamp is stored, and the UI simply does not render the link once the window passes. No data is deleted, access can be restored by updating the timestamp, and there are no edge cases around partially-cached access states.

* TapCart: The Highest-Priority Unknown

TapCart is the single largest architectural risk in this project. The brief calls out five open questions about TapCart compatibility, and until those questions are answered, the frontend architecture for all four modules cannot be finalized.

The critical ones are:

1. *Does TapCart support Shopify New Customer Accounts (passwordless auth)?* If it does not, you either need a workaround or you need to reconsider the auth architecture for the mobile surface entirely.
#+BEGIN_COMMENT
NOTE: Shopify New Customer Accounts (launched 2023) is Shopify's current default auth system --- passwordless, email OTP or magic link. TapCart was historically built against the older "classic" customer accounts (username + password). Whether TapCart has updated to support the new system must be verified directly. If it hasn't, mobile customers either can't log in, or you need a parallel mobile auth path that avoids New Customer Accounts entirely --- complicating the entire auth architecture.
#+END_COMMENT
2. *Can auth session context be passed into a WebView?* Single sign-on between TapCart and your custom platform is only possible if TapCart can pass a token or customer identifier into the WebView at load time. If it cannot, customers would need to authenticate separately in the WebView, which is a poor experience.
#+BEGIN_COMMENT
NOTE: TapCart can embed a WebView (an in-app browser) pointing at a URL on your platform. For SSO, TapCart needs to inject the customer's session token into that URL (e.g. as a query param or header) when loading the WebView. Your platform reads that token, verifies it with Shopify, and establishes a local session. If TapCart cannot do this injection, the customer lands on your platform unauthenticated and must log in again from scratch on every page load --- a broken UX.
#+END_COMMENT
3. *Does TapCart expose push notification device tokens to Klaviyo?* If TapCart manages push independently and does not expose tokens, your Klaviyo-centric push delivery plan may not work for mobile.
#+BEGIN_COMMENT
NOTE: Push notifications require a device token (a unique identifier for the customer's phone, issued by Apple/Google). TapCart manages these tokens internally for its own push system. For YOUR Klaviyo-routed notifications to reach mobile users, TapCart must expose those tokens to Klaviyo. If it manages push independently, you end up with two separate push pipelines (TapCart for commerce, Klaviyo for everything else) which creates a fragmented customer experience and duplicate notification logic.
#+END_COMMENT
4. *Which TapCart subscription would be required for this work, and how much exactly would it cost?* From preliminary research, it looks like the lower teir would not allow any custom integrations or design blocks. The next teir seems to allow for more customization, but still not enough to interface with non shopify databases. Enterprise may be the only option, and prices are not listed. We would need to nail this down.

I would want these three questions answered before scoping any frontend work. They should be verified directly with TapCart support and documentation prior to Phase 0 completing. They do not block architecture design work, but they block frontend implementation planning.

The backend foundation that every other phase depends on: the database, the API, the auth layer, the Shopify webhook pipeline, and the Klaviyo event infrastructure. On top of that foundation, the first customer-facing feature: the digital pattern library, replacing your current EDP and Uplinkly apps.

** Scope

*Platform Foundation*
- PostgreSQL database setup and schema implementation (per Phase 0 design)
#+BEGIN_COMMENT
NOTE: Provisioning the database (on DigitalOcean, either on the main droplet or as a managed DB service at ~$15/mo extra) and running migrations from the Phase 0 schema. Nothing architecturally unusual here.
#+END_COMMENT
- REST API service
#+BEGIN_COMMENT
NOTE: Your own Node.js (or equivalent) application server. Not Shopify's API. Customers and the customer portal call this; your server then calls Shopify's Admin API server-side as needed (for auth verification, fulfillment calls, catalog sync). Shopify is never contacted directly by the browser or WebView.
#+END_COMMENT
- Auth layer: Shopify New Customer Accounts integration, internal session management
#+BEGIN_COMMENT
NOTE: Middleware on every authenticated request: reads the incoming Shopify session token (passed from the frontend after the customer authenticates via Shopify's passwordless flow), verifies it with Shopify's API, extracts the Shopify customer ID, looks up or creates the internal user record, issues your own JWT. All subsequent calls use your JWT. Conceptually the same as "Login with Google" --- Shopify is just the identity provider.
#+END_COMMENT
- Shopify webhook listener service (order created, order fulfilled, fulfillment updated)
#+BEGIN_COMMENT
NOTE: An HTTP endpoint registered in the Shopify admin. Shopify sends event payloads to it. You verify the HMAC-SHA256 signature before processing. Key events for the pattern library: orders/paid (payment cleared, trigger library grant and Klaviyo email) and fulfillments/create (used for auto-fulfilling digital line items). Shopify retries delivery with exponential backoff if your endpoint doesn't return HTTP 200 within the timeout.
#+END_COMMENT
- Klaviyo event emission service, integrated at the service layer
#+BEGIN_COMMENT
NOTE: A wrapper class around Klaviyo's Track API. Named methods like trackPatternPurchased(user, pattern) keep the rest of the codebase clean and the API key isolated. Your code fires and forgets; Klaviyo handles delivery. The wrapper also lets you easily stub Klaviyo calls in tests without mocking HTTP.
#+END_COMMENT
- Deployment infrastructure (hosting, environment configuration, CI pipeline)
#+BEGIN_COMMENT
NOTE: Nginx on DigitalOcean as reverse proxy: handles SSL via Let's Encrypt, routes API and webhook paths to your Node process. PM2 or systemd keeps the app alive and restarts on crash. GitHub Actions (or similar) runs tests and deploys on merge to main. API keys and secrets stored outside the codebase in environment variables.
#+END_COMMENT

*Digital Pattern Library*
- Pattern records with file URL abstraction layer
#+BEGIN_COMMENT
NOTE: A patterns table with a file_url column. The download endpoint reads this and issues a 302 redirect. "Abstraction" means customers never get the actual hosted file URL directly --- everything flows through /api/patterns/:id/download. Admin updates file_url in the admin panel when the hosting location changes; no customer links break, no order history is invalidated.
#+END_COMMENT
- SKU-to-pattern mapping table and assignment logic
#+BEGIN_COMMENT
NOTE: A pattern_skus join table: (pattern_id, sku). One pattern maps to many SKUs (direct purchase, kit variant, bundle, legacy BigCommerce-imported SKUs). Assignment logic in the webhook handler: for each line item SKU in the incoming order, query pattern_skus. For each match, INSERT INTO user_patterns if the customer does not already own that pattern.
#+END_COMMENT
- Fulfillment logic: auto-fulfill digital line items, skip physical line items on mixed orders
#+BEGIN_COMMENT
NOTE: Shopify's Fulfillment API lets you create a fulfillment for specific line item IDs within an order, not the entire order. For a standalone digital pattern, POST to the Fulfillment API with just that line item's ID. Shopify marks it fulfilled; physical line items remain open for the warehouse. The pattern record's product_type flag (standalone vs. kit_component) controls whether this call fires at all.
#+END_COMMENT
- Historical order scan: background job to populate pattern libraries from existing Shopify order history
#+BEGIN_COMMENT
NOTE: A one-time script. Pages through all fulfilled Shopify orders using cursor-based pagination (Shopify returns a cursor in the Link response header). For each order, checks every line item SKU against pattern_skus. Inserts to user_patterns for matches. Runs at ~1 request/second to stay under the rate limit. Checkpoints progress to the DB so it resumes safely if interrupted. All grants written to an audit log. This is specifically how BigCommerce customers whose orders were migrated into Shopify get their pattern libraries restored on launch day.
#+END_COMMENT
- Customer dashboard: owned pattern library with download access
#+BEGIN_COMMENT
NOTE: A React page in the customer portal. GET /api/me/library returns the customer's user_patterns joined to patterns. UI renders each pattern with a name, thumbnail, and download button. The download button calls /api/patterns/:id/download --- server verifies ownership in the DB, reads file_url, issues the redirect. Customer never sees the actual hosted file URL.
#+END_COMMENT
- Admin panel: create/edit pattern records, manage file URLs, manage SKU mappings, manual assignment tool, view customer libraries, delivery receipt log
#+BEGIN_COMMENT
NOTE: A separate authenticated section of the frontend. Standard CRUD for patterns and pattern_skus. The manual assignment tool lets an admin look up a customer by email and add a pattern to their library directly --- covers edge cases the historical scan misses. The delivery receipt log records when each customer downloaded each pattern; useful for support queries.
#+END_COMMENT
- Klaviyo triggers: purchase confirmation with download link, pattern file updated notification
#+BEGIN_COMMENT
NOTE: Two Klaviyo events. "Pattern Purchased": fires when a user_patterns row is created, includes the download URL as a property. Klaviyo's transactional flow sends the confirmation email. "Pattern Updated": fires when admin updates a pattern's file_url. Your code queries all owners of that pattern and fires one Klaviyo event per customer. Klaviyo's flow sends the "your pattern has been updated, re-download here" notification.
#+END_COMMENT

* Phase 2 --- Comment System

** What Gets Built

A Reddit-style threaded comment system designed as a shared, reusable component. It will be integrated into the blog in this phase, and dropped into the gallery system in Phase 3 without rework.

** Scope

*Comment Engine*
- Threaded comment data model with reply depth support
#+BEGIN_COMMENT
NOTE: A comments table with a parent_id column (NULL for top-level, a comment_id for replies). The target entity (blog post or gallery submission) is stored as a polymorphic reference: target_type + target_id columns. This is what makes the component reusable --- gallery submissions pass target_type='gallery_submission' and their own ID as target_id. You can cap nesting depth (e.g. 2 levels) or allow infinite; worth confirming with the client.
#+END_COMMENT
- Upvote / downvote model
#+BEGIN_COMMENT
NOTE: A comment_votes table: (user_id, comment_id, value: +1 or -1) with a UNIQUE constraint on (user_id, comment_id) to prevent double-voting. A denormalized score column on comments is updated on each vote insert/update for fast reads, rather than aggregating votes on every query.
#+END_COMMENT
- Auto-approval for authenticated users; held queue for blocked users
#+BEGIN_COMMENT
NOTE: On comment insert, check users.status. If 'active': set comments.status='approved' immediately, visible in feed. If 'blocked': set comments.status='pending', invisible to other users, goes to moderation queue. The blocked user sees their comment as submitted but it never surfaces publicly --- they don't know they're blocked.
#+END_COMMENT
- User block and status flag (active / blocked) on customer record
#+BEGIN_COMMENT
NOTE: A status column on the users table. Admin sets it via the moderation interface. Every content submission path checks this flag. The column can be extended with more states later (shadowbanned, read-only, etc.) without changing the architecture.
#+END_COMMENT

*Notifications*
- In-app notifications for replies and upvote thresholds (not per-upvote)
#+BEGIN_COMMENT
NOTE: A notifications table: (user_id, type, reference_id, read_at). Created server-side when a reply is posted (notify the parent comment's author) or when a comment's vote count crosses a threshold (e.g. 5, 10, 25 upvotes) --- not on every individual upvote, to avoid notification spam. The customer portal polls GET /api/me/notifications for unread badge count and list.
#+END_COMMENT
- Notification preference model: per-type toggles for email vs. push, stored in platform DB, synced to Klaviyo profile properties
#+BEGIN_COMMENT
NOTE: A user_notification_preferences table: (user_id, notification_type, email_enabled, push_enabled). For example, a customer can have comment_reply: email=true, push=false, independent of upvote notifications. On each preference change, your service syncs to Klaviyo as profile properties. Klaviyo flows read these at delivery time to decide whether to actually send each notification.
#+END_COMMENT
- Klaviyo triggers routing all notification events to email and push delivery
#+BEGIN_COMMENT
NOTE: When your server creates a notification record, it simultaneously fires a Klaviyo event. Klaviyo handles routing: email? push? both? neither? All that logic lives in Klaviyo flows, which the client configures without code deployments. This is the key benefit --- adding a new notification channel (SMS, etc.) requires zero code changes on your platform.
#+END_COMMENT

*Moderation*
- Admin interface: review flagged content, manage user status, review blocked user comment queue
#+BEGIN_COMMENT
NOTE: A moderation dashboard in the admin panel. Shows comments with status='pending' (from blocked users) and comments flagged by other users. Admin can approve, delete, or escalate. Also shows a user list with block/unblock controls.
#+END_COMMENT

*Blog Integration*
- Comment component integrated into Shopify blog post pages
#+BEGIN_COMMENT
NOTE: Shopify blog pages are rendered server-side from Liquid templates (Shopify's own templating language, similar conceptually to Jinja2). You inject a mounting div into the blog post Liquid template with a data attribute containing the article ID. Your React app mounts on this div, reads the attribute, and loads comments from your API.
#+END_COMMENT

A custom gallery replacing the current Elfsight plugin and JotForm workflow, fully integrated with your Shopify product catalog and the comment system built in Phase 2.

** Scope

- Shopify product catalog sync: webhook on new product creation + nightly full sync fallback

#+BEGIN_COMMENT

* Phase 4 --- Online Event System

** What Gets Built

A custom event registration and delivery system for online workshops, seminars, and guest speaker events, integrated with Shopify for paid ticketing and your Klaviyo pipeline for all customer communications.

** Scope

*Event Management*
- Admin event creation: title, description, schedule, speaker info, capacity (with default), ticket price (or free), video link field (manually entered Zoom or equivalent link)
#+BEGIN_COMMENT
NOTE: Standard CRUD for an events table. The video link is a plain text field --- admin creates the Zoom meeting separately and pastes the URL in. No Zoom API integration. Capacity is an integer column; registration is blocked once the count of event_registrations for that event reaches it.
#+END_COMMENT
- Post-event: admin uploads VOD link and thumbnail with optional expiry date
#+BEGIN_COMMENT
NOTE: After the event, admin opens the event record and adds a vod_url (e.g. a YouTube/Vimeo unlisted video URL) and an optional vod_expiry timestamp. The customer dashboard uses these to conditionally render or hide the VOD section. Data is never deleted.
#+END_COMMENT

*Registration*
- Free events: one-click registration for logged-in customers
#+BEGIN_COMMENT
NOTE: POST /api/events/:id/register. Server checks: user is authenticated, capacity not reached, user not already registered. If all pass, inserts event_registrations row and fires the Klaviyo confirmation event. Straightforward.
#+END_COMMENT
- Paid events: registration triggers Shopify checkout; confirmation on order fulfillment webhook
#+BEGIN_COMMENT
NOTE: Each paid event has a corresponding Shopify product (a ticket product, created manually in Shopify). Clicking "Register" redirects the customer to Shopify checkout. When they complete purchase, the orders/paid webhook fires on your listener (same Phase 1 infrastructure). Your webhook handler creates the event_registrations row and fires the Klaviyo confirmation. No new infrastructure needed.
#+END_COMMENT
- Capacity enforcement with configurable cap per event
#+BEGIN_COMMENT
NOTE: There is a race condition: two customers registering simultaneously when one slot remains could both pass the capacity check before either transaction commits. The safe solution is a database-level lock: use SELECT FOR UPDATE on the event row within a transaction, or an atomic counter with a constraint. Without this you will over-enroll.
#+END_COMMENT
- Rate-limiting per account as secondary spam protection
#+BEGIN_COMMENT
NOTE: Since Shopify account login is the primary gate, the rate limit just needs to be light (e.g. max 5 registration attempts per account per hour). Implemented in API middleware. Can be stored in a DB table or Redis if added.
#+END_COMMENT
- Klaviyo trigger on registration: confirmation (email + push)
#+BEGIN_COMMENT
NOTE: "Event Registration Confirmed" Klaviyo event with properties: event title, event date as ISO timestamp, speaker name, event_id. The event_date property is what enables Klaviyo's time-based reminder flows --- Klaviyo can trigger "send 1 day before event_date" entirely within their flow builder, with zero additional code on your end.
#+END_COMMENT
- Klaviyo-driven reminder flow leading up to event
#+BEGIN_COMMENT
NOTE: This is NOT a code task. It is a Klaviyo flow the client configures themselves in Klaviyo's flow builder. Your only responsibility is that the event_date is present as an ISO timestamp in the Klaviyo event payload at registration time. Klaviyo handles the time-delay trigger logic.
#+END_COMMENT

*Customer Dashboard*
- Registered events listed with event details, schedule, speaker info
#+BEGIN_COMMENT
NOTE: GET /api/me/events returns the customer's event_registrations joined to events. Standard list/card UI.
#+END_COMMENT
- Zoom link surfaces in dashboard within a defined pre-event window
#+BEGIN_COMMENT
NOTE: The API only includes the live_link field in the response if the current timestamp is within a configured window before the event start time (e.g. 30 minutes). Outside that window, the field is omitted from the response entirely. Prevents link sharing or premature access.
#+END_COMMENT
- Post-event: VOD link and thumbnail displayed while within access window
#+BEGIN_COMMENT
NOTE: Conditional API response: if event_end_time < now AND (vod_expiry IS NULL OR vod_expiry > now), include vod_url and thumbnail. After expiry, those fields are absent from the response. No separate "is_expired" flag needed --- the timestamps are the truth.
#+END_COMMENT
- VOD expiry: soft-hide (link stops rendering when expiry timestamp passes, data retained, timestamp updateable by admin)
#+BEGIN_COMMENT
NOTE: The vod_url and vod_expiry stay in the DB permanently. The conditional response logic handles the visibility. Admin extends access by updating vod_expiry in the admin panel. No data is deleted; no access is revoked at the hosting level unless the client also removes the video from YouTube/Vimeo separately.
#+END_COMMENT
- Notification when VOD is posted for an event the customer attended
#+BEGIN_COMMENT
NOTE: When admin saves a VOD URL to an event record, your API fires a Klaviyo event: "Event VOD Available." Since you need to notify all registered attendees, your code runs: SELECT user_id FROM event_registrations WHERE event_id = ? and fires one Klaviyo event per customer. Klaviyo delivers per each customer's notification preferences.
#+END_COMMENT

** Out of Scope for This Phase

- Premium VOD packages and access tier gating (flagged in brief as future scope: agreed, best handled once the base system is running)
#+BEGIN_COMMENT
NOTE: The brief mentioned a future upsell: a purchasable tier that unlocks all past event recordings. This requires an access_tier column on the users table and gating logic on VOD endpoints. Not complex to add later, but needs the base event system running first. Worth leaving a nullable access_tier column on users in the Phase 1 schema so it doesn't require a migration later.
#+END_COMMENT
- Zoom API integration (not needed; link is a manually entered field)
#+BEGIN_COMMENT
NOTE: Zoom has an API that could auto-create meetings when you create events. The brief explicitly does not want this. Admin creates the Zoom meeting manually and pastes the link in. Keeps scope clean and removes a dependency on Zoom's API pricing and terms.
#+END_COMMENT

$11,000 -- $17,000
